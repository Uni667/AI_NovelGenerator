# Memory 状态系统架构解析

在传统 AI 写作工具中，生成引擎通常只依靠前几章全文或固定的粗略大纲来进行提示。这在长篇小说连载中会导致严重的“幻觉”和“设定遗忘”。

本项目的核心护城河是存在于项目文件夹下 `memory/` 目录的状态系统。

## 1. 核心状态文件 (Merged State)

所有已确认的、官方的小说事实被存储为高度结构化的 JSON，这些文件构成大模型生成章节时的上下文底座。

- `global_summary.md`: 小说的当前故事总摘要，描述迄今为止到底发生了什么。
- `character_state.json`: 所有人物的生死、位置、动机、揭秘情况和相互关系的实时切片。
- `name_usage_rules.json`: 解决网文中常见的“隐藏身份”、“多重马甲”、“特定人员称呼不同”问题。确保旁白和不同角色的对话用词准确。
- `plot_threads.json`: 伏笔追踪器，记录埋下的伏笔 (active) 和已收束的伏笔 (resolved)。
- `outline_state.json`: 取代了原有的静态 `Novel_directory.txt`。动态记录已定稿章节目标和未来未写章节的大纲安排。

## 2. 隔离区：补丁与差异化 (Pending State)

AI 不是人，AI 会产生幻觉。所以 AI 提取的设定绝不能立刻写入上述的核心状态文件中。

- `memory/patches/`: 当章节定稿后，AI 会提取 `State Patch` 放在这里。它们的状态标记为 `pending_review`。此时它们被**隔离**，不会被注入到下一章的生成上下文中。只有当你点击“合并 (Merge)”后，系统才会解析这个补丁，增量应用到上述五大状态文件中。
- `memory/outline_diffs/`: 当进行大纲演化时，生成的演化建议放在这里。只有确认“应用”后，才会更新 `outline_state.json`。

**数据流向总结**：
`草稿定稿` -> `AI 生成 Patch` -> `放在隔离区` -> `人工确认合并` -> `更新长期记忆` -> `影响后续生成`

## 3. 高风险管控 (Risk Level)

AI 有时会犯非常离谱的错误（例如：让主角死亡、把伏笔全部删空）。
在 Patch 和 Diff 系统中，我们实现了**高风险诊断**。
如果 AI 提取的动作涉及：修改人物真名、改变生死、修改大纲关键走向等，该 Patch 会被系统强行标记为 `high` (高风险)。
在工作台和状态总览页，红色的警告会始终提示作者优先处理这些高风险修改。

## 4. 为什么不依赖 Novel_directory.txt 了？

旧系统把大纲直接读写成 TXT，这很容易在后续演化中因大模型格式错乱而全盘崩溃，导致项目不可逆损坏。
现在，主大纲存储在安全的 `outline_state.json` 中。旧的 `Novel_directory.txt` 仅作后向兼容，在体验优化版本 (Phase 9) 后不再作为系统的主要驱动数据。
